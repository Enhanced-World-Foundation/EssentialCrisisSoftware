## PART 1 — PROJECT DESCRIPTION (~2000 characters)

```
Essential Crisis Software (ECS) is an open-source, offline-first mesh
framework for local communication and coordination when network
infrastructure is unavailable, degraded, or hostile. It is not a single
app but a modular system: a build package per scenario, a core framework
handling orchestration, I/O gating and process masking, functional
modules (communication, groups, routing, SOS, map, note-board,
reputation), and a plugin layer where cryptographic and RF backends swap
at runtime.

ECS targets four scenarios with distinct threat models. A — Dead-Internet:
ISP/backbone failure, no active adversary; the problem is discovery and
local routing. B — Fallout: long-term grid collapse, hardware and power
scarcity, proxy identity, Byzantine fault tolerance. C — Total War:
state-grade adversary with automated spectrum analysis and AI signal
classification; anti-detection at signal and topology level. D — Hybrid
War: mixed conditions, surveillance state, jamming, infiltration —
configurable presets.

The MVP runs on Bluetooth Low Energy, present in every phone, license-free
and low-power: open/closed groups, network identification, opportunistic
relay, compressed messaging, local forum, SOS broadcast, a tiered
note-board with emergency pins, and local maps. MVP 1.1 adds
encryption/steganography, alternative transmission media (WiFi mesh,
LoRa/Meshtastic, HF NVIS), an offline barter/reputation system, and
non-satellite navigation.

Transmission tiers by range: WiFi locally, LoRa for metro distances, HF
NVIS for regional reach — each matched to a power budget and threat
profile.

ECS is pro-bono. It cannot be commercialized — there is no market for
infrastructure built for civilizational failure — but it should exist.
The repo is open so the design is audited, not trusted. We need protocol,
embedded, RF and cryptography engineers who want a hard, real problem
without commercial pressure.

GitHub: github.com/TianLongLun/EssentialCrisisSoftware
```

## PART 2 — SCENARIO & TECH ANALYSIS

### 2.1 How the MVP maps to scenarios

The MVP is not scenario-neutral — it is **Scenario A.1 hardware** (blackout, days→months, no active adversary). That is the correct first target: largest addressable need, lowest security burden, every phone is a node.

| Build | Scenario fit | Adversary | Security burden | Why build first/later |
|-------|-------------|-----------|-----------------|----------------------|
| **MVP** | A.1 (blackout, days–months) | None | Minimal | Ships fastest, proves the mesh works, recruits contributors |
| **MVP 1.1** | A.2 + D.1 (prolonged, hybrid, light chaos) | Passive-mass + targeted | Medium | Adds the layers an adversary forces: crypto, alt-media, reputation |
| **A full** | A.1+A.2 | None→low | Low | Hardened, documented, multi-medium |
| **B** | Fallout | Fragmented/local | High (physical+power) | Needs power plans + proxy identity; depends on A maturity |
| **C** | Total War | State + AI | Extreme (anti-detection) | Hardest; classified modules; build last, on proven base |
| **D** | Hybrid | Mixed | Configurable | Preset layer over A/B/C — emerges, not built standalone |

**Assumption being made explicit:** MVP usefulness assumes node *density*. BLE mesh only works if enough phones run it in range of each other. Below a density threshold the mesh is islands, not a network. This is the single biggest adoption risk and should be stated in docs, not hidden.

---

### 2.2 Transmission media — honest tech indications

Ranges are **condition-dependent approximations**, not specs. Real numbers depend on terrain, antenna, interference, and hardware. Treat as planning estimates, validate on hardware.

| Medium | Realistic range | Throughput | Power draw | License | Detectability | Primary scenario |
|--------|-----------------|-----------|-----------|---------|---------------|------------------|
| **BLE** | 10–100 m; BLE5 coded PHY a few hundred m LOS `[EST]` | Low | Very low | None | Low–moderate | MVP / A |
| **WiFi direct/mesh** | 50–250 m typical; ~1–2 km only with directional antenna + clear LOS `[OPTIMISTIC]` | High | High | None | High (loud RF) | A, dense urban |
| **LoRa / Meshtastic** | 2–10 km, terrain-dependent; rural LOS can exceed, urban often 2–5 km `[EST]` | Very low (text/telemetry) | Low | ISM (868/915 MHz, free) | Moderate | A inter-area, B |
| **HF NVIS (40/80 m)** | Regional, ~0–400 km skywave `[EST]` | Very low, slow digital | High + antenna | Ham (moot in crisis) | High — direction-findable | B regional |
| **Optical / acoustic / LOS** | Short, directional | Low | Low | None | Very low (no broad RF) | C stealth `[EXPERIMENTAL — needs research]` |

**Your stack progression is sound:**
```
WiFi mesh (local)  →  LoRa (metro/inter-town)  →  HF NVIS (regional)
```
Each tier trades bandwidth and power for range. The right framing in docs: *not* "which is best" but "which is appropriate for this scenario's range need and threat level."

### 2.3 Tech indication per scenario

| Scenario | Dominant constraint | Media that fit | Media to avoid | Power plan | Crypto/stealth need |
|----------|--------------------|--------------|---------------|-----------|---------------------|
| **A.1** blackout | Convenience, density | BLE, WiFi | — | Grid mostly up; phone batteries | Optional |
| **A.2 / D.1** prolonged/hybrid | Targeted threats appear | BLE, WiFi, LoRa | — | Intermittent grid; need charge plan | Encryption + obfuscation **on** |
| **B** fallout | Power + hardware scarcity, range | LoRa (low power), HF (regional) | High-power WiFi | Solar, undervolt, max-runtime, SOS over-voltage | Proxy identity, key mgmt without PKI |
| **C** total war | Detection = death | Optical/acoustic, burst LoRa, minimal RF | HF (DF-able), constant WiFi | Cold/dark operation, minimal emission | Anti-detection dominates; signal masking, freq hopping, topology churn |
| **D.2** custom | User-defined | Configurable | Per config | Per config | Per config |

**Key engineering tension to flag for contributors:** the features that make A.1 useful (loud discovery, persistent beacons, easy pairing) are exactly what gets you found in C. These are *opposite* design goals. ECS resolves this by scenario-specific build packages, not one universal client. That is the core architectural justification — worth stating explicitly when recruiting, because it's the kind of constraint good engineers respect.

### 2.4 MVP feature usefulness ranking (A.1 context)

| Feature | Usefulness in A.1 | Build difficulty | Verdict |
|---------|------------------|-----------------|---------|
| SOS broadcast | Critical | Low | MVP core |
| Open/closed groups | High | Low | MVP core |

| Opportunistic relay/bypass | Critical (this *is* the mesh) | High | MVP core |
| Identification (custom/standard ID) | High | Medium | MVP core |
| Reputation/barter | Low in A.1, high in A.2/B | High | **Defer to 1.1** |
| Non-GPS navigation | Low in A.1, high in B/C | High | **Defer to 1.1** |
| Encryption/stego | Low in A.1, critical in 1.1+ | High | **Defer to 1.1** |

**Recommendation:** the MVP scope you wrote is correct but slightly heavy. Map data and relay logic are the two hard pieces — everything else is UI over a working mesh. If timeline slips, the irreducible MVP is: **relay + groups + compressed messaging + SOS**. Map and note-board can be the first post-MVP additions. Stated plainly so you can cut scope under pressure without losing the core.
